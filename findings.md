# Результаты исследований: Технические открытия и ограничения

## Проблема с верификацией карты на Koyeb
- **Наблюдение**: При регистрации на платформе Koyeb.com запрашиваются данные банковской карты для активации бесплатного тарифа.
- **Решение**: Koyeb не подходит для бесплатного использования без ввода карты. Проект переведен на Hugging Face Spaces + Neon.tech.

## Альтернативы хостинга без привязки карт

### 1. PythonAnywhere
- **Описание**: Облачный хостинг для Python-приложений. Бесплатный тариф не требует карты.
- **Ограничения**: Доступны только запросы из белого списка. Запуск фоновых процессов («Always-On») на бесплатном тарифе ограничен.

### 2. Hugging Face Spaces (Docker-контейнер)
- **Описание**: Платформа для развертывания веб-приложений и Docker-контейнеров. Бесплатный тариф CPU не требует карты.
- **Плюсы**: Позволяет боту работать непрерывно (always-on).
- **Минусы**: Файловая система временная (данные стираются при перезапусках контейнера).

## Базы данных без привязки карт
- **Neon.tech (Serverless PostgreSQL)**: Предоставляет полноценную бесплатную базу данных PostgreSQL без необходимости привязывать карту. Идеально подходит для сохранения подписок бота.

## Особенности работы с внешними API

### 1. Bybit REST API
- **Эндпоинт**: `https://api.bybit.com/v5/market/tickers?category=linear&symbol=<SYMBOL>`
- **Доступ**: Публичный эндпоинт, ключи авторизации для чтения котировок не требуются.

### 2. Технический анализ TradingView (библиотека `tradingview-ta`)
- **Библиотека**: `tradingview-ta`
- **Описание**: Получает сводные рекомендации (BUY, SELL, NEUTRAL) и точные значения индикаторов (RSI, MA) напрямую через публичный сканер TradingView без использования API-ключей.

## Ошибки интерфейса и обработки сообщений
- **Наблюдение**: Переход к бесшовному обновлению сообщений (Clean Screen) привел к исключениям `NameError: name 'edit_rich_message' is not defined` в `features/bot.py`.
- **Решение**: Добавлена функция `edit_rich_message`, использующая метод `bot.edit_message_text` с HTML-парсингом и безопасным откатом к тексту без HTML-тегов при возникновении ошибок разметки.
- **Наблюдение**: При добавлении `@bot.middleware_handler` без предварительного включения поддержки middleware библиотека pyTelegramBotAPI выбрасывала ошибку `RuntimeError: Middleware is not enabled. Use apihelper.ENABLE_MIDDLEWARE before initialising TeleBot.`
- **Решение**: Добавлена строка `telebot.apihelper.ENABLE_MIDDLEWARE = True` перед инициализацией `bot = telebot.TeleBot(TOKEN)`.
- **Наблюдение**: Реализация Clean Screen (Чистый Чат) сохраняла `last_message_id` в таблице `subscriptions` для каждого активного чата. Поскольку подписка на сигналы определялась простым наличием строки для `chat_id` в этой таблице (`SELECT 1 FROM subscriptions`), каждый пользователь при первом же действии (даже без согласия) автоматически подписывался на сигналы и не мог отписаться (так как отправка подтверждения отписки снова создавала запись в БД с новым `last_message_id`).
- **Решение**: Добавлен отдельный флаг `signals_subscribed BOOLEAN DEFAULT FALSE` в схему таблицы `subscriptions`. Переписаны функции `subscribe_user`, `unsubscribe_user`, `is_subscribed` и `get_all_subscribers` для работы с этим флагом. Теперь `last_message_id` отслеживается без автоматической принудительной подписки на сигналы.
- **Наблюдение**: В методах голосового обработчика и обработки текстовых тикеров использовалась функция `bot.reply_to(message, ...)`. Это приводило к сбою `telebot.apihelper.ApiTelegramException: Bad Request: message to be replied not found`, так как входящее сообщение пользователя удалялось в middleware до начала работы основного обработчика.
- **Решение**: Вызовы `bot.reply_to` заменены на `bot.send_message(message.chat.id, ...)`, что исключает зависимость от существования удаленного сообщения пользователя.
- **Наблюдение**: Вывод логов библиотеки `TeleBot` об ошибках поллинга и предупреждения `FutureWarning` о переходе на `google.genai` приводили к выводу большого количества посторонних трейсбеков в консоль и отображению предупреждений как ошибок в IDE.
- **Решение**: Настроено игнорирование предупреждений `FutureWarning` во всех задействованных файлах. Установлен уровень логирования `telebot.logger.setLevel(logging.CRITICAL)` для исключения отображения трассировок при кратковременных сетевых сбоях.





## Утечка токена Telegram-бота
- **Наблюдение**: Старый токен Telegram-бота был жестко прописан в публичном файле `README.md` на строке 57 и попал на GitHub.
- **Решение**: Токен в `README.md` заменен на безопасный плейсхолдер `YOUR_TELEGRAM_BOT_TOKEN`. Новый токен прописан исключительно в локальном файле `.env`, который добавлен в `.gitignore` и не отслеживается Git.

## Интеллектуальное распознавание тикеров из голоса
- **Наблюдение**: Голосовой обработчик поддерживал только 6 жестко закодированных монет, из-за чего запросы других тикеров (например, Tron/TRX) завершались ошибкой.
- **Решение**: Разработан метод `extract_ticker_from_text`, сочетающий быстрый поиск по локальному словарю популярных коинов (30+ монет) и отказоустойчивое AI-распознавание (Llama/Gemini) для редких тикеров. Теперь бот корректно извлекает любой тикер из голосового запроса.

## Реализация Clean Screen (Чистый Чат)
- **Наблюдение**: Накопление истории входящих и исходящих сообщений в приватном чате приводило к образованию длинной "простыни", ухудшая UX.
- **Решение**: Реализовано автоматическое удаление входящих сообщений пользователя с помощью Telegram Middleware. Все исходящие сообщения отслеживаются через обёртку `bot.send_message` и сохраняются в PostgreSQL. При каждом новом запросе предыдущие сообщения бота удаляются, оставляя чат абсолютно чистым (активным остается только последнее актуальное меню или сигнал).

## Настройка MCP-сервера Context7
- **Наблюдение**: При попытке запустить Context7 через CLI/npx локально выяснилось, что в текущем окружении терминала отсутствуют Node.js и npx (команды не распознаны).
- **Решение**: Настроен удаленный запуск сервера через защищенное SSE-подключение к официальному эндпоинту `https://mcp.context7.com/mcp` в конфигурационном файле `mcp_config.json`. Это позволило полностью исключить необходимость локальной установки Node.js и npx. Для расширения лимитов запросов успешно интегрирован предоставленный пользователем API-ключ авторизации.

## Консенсус-алгоритм 2.0 и Обход блокировок CloudFront
- **Наблюдение**: Прокси-сервер на Cloudflare Worker при запросе свечей (`/fapi/v1/klines`) возвращает ошибку 403 Forbidden от CloudFront из-за ограничений безопасности Binance для облачных IP-адресов. Однако прямые запросы к API Binance (`fapi.binance.com`) и Bybit (`api.bybit.com`) с локальной машины пользователя проходят успешно с кодом 200.
- **Решение**: Переработана сетевая архитектура получения свечей и котировок в `market_data.py` по принципу "прямое подключение -> каскад прокси". Если прямое подключение к Binance заблокировано (например, при развертывании в облаке Hugging Face), бот последовательно опрашивает прокси Binance, затем прямой API Bybit (с реверсом массива свечей для сохранения хронологии) и прокси Bybit. Это гарантирует 100% стабильность получения данных в любых средах хостинга.
